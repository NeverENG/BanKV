# 客户端与服务端 TLV 协议交互方案

## 项目现状分析

### 当前结构
- **服务端**：基于 Raft 共识的 KV 存储服务
- **FSM**：已实现，负责 Raft 日志应用到存储
- **存储**：基于 MemTable 的 KV 存储引擎

### 缺失部分
- **网络通信**：服务端未实现网络监听
- **协议处理**：未定义客户端与服务端的通信协议
- **客户端**：未实现客户端程序

## TLV 协议设计

### 1. TLV 格式定义

```
+--------+--------+--------+----------------+
| 类型   | 长度   | 版本   | 数据           |
+--------+--------+--------+----------------+
| 4字节  | 4字节  | 4字节  | 可变长度       |
+--------+--------+--------+----------------+
```

### 2. 消息类型定义

| 类型值 | 消息类型 | 说明 |
|--------|----------|------|
| 1      | PUT      | 写入键值对 |
| 2      | GET      | 获取值 |
| 3      | DELETE   | 删除键 |
| 4      | RESPONSE | 响应消息 |
| 5      | ERROR    | 错误消息 |

### 3. 数据结构

#### PUT 消息
```
+--------+--------+--------+--------+--------+--------+----------------+----------------+
| 类型=1 | 长度   | 版本=1 | key_len| value_len | key_data | value_data |
+--------+--------+--------+--------+--------+--------+----------------+----------------+
```

#### GET 消息
```
+--------+--------+--------+--------+----------------+
| 类型=2 | 长度   | 版本=1 | key_len| key_data       |
+--------+--------+--------+--------+----------------+
```

#### DELETE 消息
```
+--------+--------+--------+--------+----------------+
| 类型=3 | 长度   | 版本=1 | key_len| key_data       |
+--------+--------+--------+--------+----------------+
```

#### RESPONSE 消息
```
+--------+--------+--------+--------+--------+----------------+
| 类型=4 | 长度   | 版本=1 | status | value_len | value_data |
+--------+--------+--------+--------+--------+----------------+
```

#### ERROR 消息
```
+--------+--------+--------+--------+--------+----------------+
| 类型=5 | 长度   | 版本=1 | code   | msg_len | msg_data       |
+--------+--------+--------+--------+--------+----------------+
```

## 技术要点

### 1. 二进制协议优势
- **高性能**：减少网络传输量
- **低开销**：无需序列化/反序列化
- **跨语言**：可被不同语言实现

### 2. 安全考虑
- **消息长度限制**：防止缓冲区溢出
- **类型检查**：确保消息类型有效
- **错误处理**：优雅处理异常情况

### 3. 性能优化
- **连接池**：复用 TCP 连接
- **批量操作**：合并多个请求
- **异步处理**：非阻塞 I/O
